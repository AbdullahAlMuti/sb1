# Plan: Fix "item specific" errors in Amazon→eBay listing (match SuperDS smoothness)

## Context

The extension scrapes Amazon and programmatically creates an eBay listing draft, then
redirects the user to eBay's draft editor. On the editor page, eBay highlights **required
item specifics as missing/invalid and blocks Publish** — on both single-item and
multi-variation listings. SuperDS (which our pipeline was originally ported from) does this
smoothly.

Root cause is in the item-specifics (aspect) mapping, not the auth/draft/EPS plumbing
(those work — the save returns 200, the editor opens). The current mapping in
`updateListing` and `addVariations` ([common/ebay-listing-api.js](apps/extension/common/ebay-listing-api.js)):

1. **Guesses eBay's attribute-schema field names.** Required detection tries
   `attr.required || attr.isRequired || attr.usage === 'REQUIRED' || attr.usageConstraint
   === 'REQUIRED' || attr.minValues>0 || attr.groups…` ([ebay-listing-api.js:264-289](apps/extension/common/ebay-listing-api.js#L264)).
   Only `attributeName` is confirmed real; the rest are unverified. If the real field name
   differs, required aspects are never detected → never filled → editor flags them.
2. **Does no value validation against eBay's allowed-value list.** Scraped free-text spec
   values are sent as-is. For SELECTION_ONLY aspects, an off-list value is rejected.
3. **Uses unsafe defaults for constrained aspects.** Falls back to `'Does not apply'` /
   `attr.options[0].value`, but `options` may be the wrong field, and `'Does not apply'` is
   invalid for a SELECTION_ONLY aspect.
4. **Variation path (`addVariations` → `msku-update`) carries no item-specifics at all** —
   it only sends `variationSpecific` dimensions, so category-level required aspects on a
   multi-variation listing are always empty.
5. **Category pick is crude** — first leaf of the keyword search ([ebay-listing-api.js:1221-1226](apps/extension/common/ebay-listing-api.js#L1221)).
   Wrong category → wrong required-aspect set.

**Intended outcome:** an Amazon product (single or multi-variation) lists to eBay with all
required item specifics populated with valid values, so the draft editor opens clean and
Publish is not blocked — matching SuperDS.

> Note: `apps/extension/` is the source of truth. Never edit
> `apps/web/public/chrome_extension/` (generated by `prepare:dev`/`prepare:prod`).

## Phase 1 — `aspect-mapper.js`: a pure, tested item-specifics builder

New module `apps/extension/common/aspect-mapper.js` exposing `window.SSAspectMapper`. Pure
functions (no chrome/DOM/network) so it is unit-testable with `node:test`, following the
existing test pattern ([tests/listing-draft.test.js](apps/extension/tests/listing-draft.test.js),
`tests/helpers/load-global.js`).

API:
```
buildAspects(attributeList, scrapedSpecs, opts) -> { aspectName: [values...] }
```
Behavior (using the **verified** field names from Phase 0):
- **Required detection** from the real schema field (replaces the guess-OR chain).
- **Selection-only vs free-text** per aspect. For selection-only, normalize the scraped
  value and match (case/space/singular-plural-insensitive — reuse the logic already in
  `matchAspectName` [ebay-listing-api.js:465](apps/extension/common/ebay-listing-api.js#L465))
  against the aspect's allowed-value list; only emit if it matches.
- **Safe defaults for required-but-unfilled:** selection-only → first allowed value;
  Brand (free-text/custom-allowed) → `'Unbranded'`; other free-text → `'Does not apply'`.
  Never emit `'Does not apply'` for a selection-only aspect.
- **Drop unknown aspects** not present in `attributeList` (eBay rejects/ignores them and
  they can trip validation).
- Respect `maxValues`/multi-select (split on comma only when allowed).

Reuse `matchAspectName`'s synonym map (itemModelNumber→mpn, manufacturer→brand, …) — extract
it into the new module so both single and variation paths share one source of truth.

**Acceptance:** given the Phase-0 `attributeList` fixture + a scraped specs object,
`buildAspects` fills every required aspect with a valid value, validates selection-only
values, and drops unknowns.

**Verification:** `node --test apps/extension/tests/aspect-mapper.test.js` passes; cases
cover required-fill, selection-only match + reject, free-text passthrough, Brand default,
unknown-drop, multi-select.

## Phase 2 — Wire the mapper into both upload paths

- **Single path:** replace the inline spec-map + required-fill in `updateListing`
  ([ebay-listing-api.js:247-289](apps/extension/common/ebay-listing-api.js#L247)) with a
  single `window.SSAspectMapper.buildAspects(attributeList, product.prod_specs, …)` call.
  Payload shape unchanged otherwise.
- **Variation path:** in `addVariations`
  ([ebay-listing-api.js:520](apps/extension/common/ebay-listing-api.js#L520)), the
  `msku-update` body currently sends only `variationSpecific` dimensions and no
  category-level item specifics. Add the mapper output as the listing-level aspects so
  required category aspects (Brand/Type/etc.) are populated for multi-variation listings
  too. Confirm against the Phase-0 SuperDS multi-variation fixture which field carries
  them (listing-level `attributes` vs a separate field) and match it exactly.
- Register `common/aspect-mapper.js` in `manifest.json` + `manifest.prod.json` content-script
  arrays **before** `common/ebay-listing-api.js` (load order matters — mirror how
  `SSSkuEngine`/`SSVariationNormalizer` are ordered).

**Acceptance:** both single and multi-variation real uploads produce a draft whose editor
shows **no** missing/invalid required item specifics.

**Verification:** list one single + one multi-variation Amazon product; the eBay draft
editor opens with required item specifics pre-filled and Publish enabled. Diff our new PUT
payload against the SuperDS fixture from Phase 0 — required aspects now present.

## Phase 3 — Category accuracy + post-save safety net

- **Category:** improve selection beyond "first leaf"
  ([ebay-listing-api.js:1221-1226](apps/extension/common/ebay-listing-api.js#L1221)) — prefer
  the highest-ranked leaf eBay returns, and feed brand/category hints from the scrape into
  `getCategoryRecommendations` so required aspects belong to the right category. Compare the
  chosen `categoryId` to the one SuperDS lands on (visible in the Phase-0 capture URL).
- **Safety net (optional, if errors persist):** after `saveListing`, re-read the draft and,
  if any required aspect is still empty, fill from defaults and re-save once. Keeps the
  editor clean even when the scrape had no usable value.

**Acceptance:** category matches SuperDS for the test products; no required aspect is left
empty after upload.

**Verification:** re-run the two test products; categoryId matches the SuperDS capture;
editor clean.
