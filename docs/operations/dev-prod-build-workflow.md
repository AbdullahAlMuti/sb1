# SellerSuit Dev, QA, and Production Workflow

SellerSuit is currently eBay-only. Shopify exists in the repository but is intentionally disabled and is future scope only.

## Folder Map

| Area | Path | Purpose |
| --- | --- | --- |
| Web dashboard | `apps/web` | eBay dashboard, auth, billing, listings, templates, extension connect |
| Chrome extension | `apps/extension` | Manifest V3 source of truth, side panel, supplier import, eBay upload |
| Admin | `apps/admin` | Admin operations UI |
| Marketing | `apps/marketing` | Public marketing site |
| Shared packages | `packages/*` | Auth, API client, UI, types, config, utilities |
| Backend | `supabase/functions` | Edge Functions for eBay, billing, auth, extension, admin |
| Database | `supabase/migrations` | Schema, RLS, RPCs, and database state |
| Dev extension output | `apps/extension/dist/extension-dev` | Local unpacked Chrome extension |
| Prod extension output | `apps/extension/dist/extension-prod` | Chrome Web Store source artifact |
| Public extension copy | `apps/web/public/chrome_extension` | Generated from the production extension build |

Do not edit `apps/web/public/chrome_extension` directly. It is generated by `npm run prepare:extension:prod`.

## Local-First Feature Rule

All new features must be added and tested in the local/dev environment first.
Use the local web/admin servers, local or non-production backend targets, and
`apps/extension/dist/extension-dev` for Chrome extension testing. Prepare or
publish production artifacts only after the local/dev flow is verified.

This is a permanent release gate, not an optional convention. Root production
commands run the local/dev verification path first, and the workflow checker
fails if that command order is changed.

## Environment Mapping

| Surface | Local | Production |
| --- | --- | --- |
| Marketing | `http://localhost:3000` via `MARKETING_APP_URL` / `VITE_MARKETING_URL` | `https://sellersuit.com` |
| Web dashboard | `http://localhost:3001` via `APP_URL` / `PUBLIC_APP_URL` / `VITE_APP_URL` | `https://sellersuit.com` |
| Admin | `http://localhost:3002` via `ADMIN_APP_URL` / `VITE_ADMIN_URL` | production admin URL |
| Supabase | `http://127.0.0.1:54321` or staging via `SUPABASE_URL` / `VITE_SUPABASE_URL` | production Supabase URL |
| Functions API | `<SUPABASE_URL>/functions/v1` | `<SUPABASE_URL>/functions/v1` |
| Extension manifest | `apps/extension/manifest.dev.json` | `apps/extension/manifest.prod.json` |
| Extension config | `apps/extension/common/config.dev.js`, then `.env.local` rewrite | `apps/extension/common/config.prod.js` |

`WEB_APP_URL` maps to the existing app URL variables. `API_BASE_URL` maps to the Supabase Functions base URL derived from `SUPABASE_URL`; do not hardcode it separately.

## Local Workflow

```bash
npm run analyze:project
npm run dev:local
npm run qa:local
```

`npm run dev:local` starts the extension watcher and local app servers:
marketing on `http://localhost:3000`, dashboard on `http://localhost:3001`,
and admin on `http://localhost:3002`. Load `apps/extension/dist/extension-dev`
as an unpacked Chrome extension. To run only the extension watcher, set
`SELLERSUIT_EXTENSION_ONLY=true` before `npm run dev:extension`.

## Production Workflow

```bash
npm run release:ready
```

This runs `qa:local` first, then production extension preparation, production
extension verification, and the production workflow artifact scan. If local/dev
verification fails, no production artifact should be treated as releasable.

Use `apps/extension/dist/extension-prod` or the generated production zip from `apps/extension/dist` for Chrome Web Store packaging.

## Targeted Commands

```bash
npm run check:workflow:local
npm run check:workflow:prod
npm run check:extension
npm run prepare:extension:dev
npm run prepare:extension:prod
npm run release:extension
```

`npm run prepare:extension:prod` also prepares and verifies
`apps/extension/dist/extension-dev` before it creates
`apps/extension/dist/extension-prod`. This keeps direct production preparation
aligned with the local-first rule.

## QA Checklist

- Dashboard: auth, signup, logout, session persistence, dashboard redirects, settings, billing, subscription, templates, extension connect.
- Backend: CORS, auth status, protected functions, eBay functions, billing functions, listing generation, template sync, admin protected routes.
- Extension: popup, side panel, auth buttons, session bridge, supplier cards, Amazon/Walmart/AliExpress import, eBay listing upload, image upload, calculator, bulk runner.
- Production artifact: no `localhost`, no `127.0.0.1`, no dev manifest, no `.env`, no debug mode, no `<all_urls>`, no test files.

## Watch Points

- `chrome.sidePanel.open()` must be triggered from a valid user gesture.
- Extension auth failures usually come from dashboard bridge origins, token storage, or the `auth-status` function.
- `manifest.prod.json` and `common/config.prod.js` must stay free of local URLs.
- `apps/web/public/chrome_extension` must only be regenerated from the production extension build.
